1 · Motivation & Goal

Goal: inject high-level reasoning into a track-conditioned video-to-video generator so that it understands scene, motion, and caption — not just raw pixels and coordinates.

Pipeline

V_s — source video (appearance + scene) T_s / T_t — source / target track coordinates C_t — target caption

Three gaps in the base generator

① No scene understanding

V_s is consumed as raw pixels. The generator has no concept of which objects exist or how the scene is structured.

② No motion semantics

Coordinate changes in T_s→T_t are followed blindly — no understanding of camera vs. object motion, zoom, rotation, etc.

③ No C_t ↔ T_t alignment

C_t and T_t are treated as independent inputs, even though they describe the same physical event.

Three capabilities VLT must learn

Object & Scene Grounding

Which object/region does a given track belong to?

"this track → person's head" · "this track → background tree"

Motion Semantics

What kind of motion does T_s→T_t represent, in what direction and speed?

"camera pans right" · "object moves forward quickly" · "zoom-in"

Caption-Track Alignment

Does a phrase in C_t correspond to a motion pattern in T_t? Are they consistent?

"C_t says move right, T_t moves left → mismatch"

VLT compresses these three into a representation \(R\), which is injected into the generator's attention layers to make generation semantically coherent.

2 · Training Tasks

VLT is a multi-task model sharing one backbone. Each task is selected by a learnable special token prepended to the input, which also routes to the correct output head.

Special tokens & task routing

TokenTaskHead
<PREDICT_TRACK>A' — predict target track coordinatesTrack Regression Head
<DESCRIBE_MOTION>B — generate motion descriptionLanguage Head
<MOTION_QA>C — answer motion-related questionsLanguage Head

Task definitions

A' PREDICT_TRACK

In: (V_s, T_s, C_t) or (V_s, T_s, instruction)

Out: T_t (N×T×2)   Loss: Huber/L1

Forces the model to ground the track to an object in V_s and understand how C_t maps to a coordinate-level motion.

B DESCRIBE_MOTION

In: (V_s, T_s, T_t)

Out: motion description text   Loss: CE

Aligns motion semantics into language space.

"The person moves right while the background stays static."

C MOTION_QA — three sub-types

In: (V_s, T_s, T_t, C_t, Q)   Loss: CE

  • C-obj — "What object does this track belong to?" → Grounding
  • C-motion — "Camera or object motion?" / "Direction?" → Motion Semantics
  • C-align — "Does C_t match T_t?" → Caption-Track Alignment

3 · Data Design

Sample structure — 6-tuple

(V_s, T_s, C_s,  V_t, T_t, C_t)
 source ──────   target ───────

V_s / V_t : video  (T, H, W, 3)   — same appearance, different motion
T_s / T_t : track  (N=64, T, 2)   — N: number of point, 2 : pixel coordinates
C_s / C_t : caption (text)

Automatic instruction generation

C_s: "A person walking slowly to the left in a park"
C_t: "A person running quickly to the right in a park"
                         ↓ LLM
  instruction: "Make the person move faster and change direction to the right"

Fallback: if C_s ≈ C_t, derive template from ΔT = T_t − T_s

Track volume rasterization

Sparse track (N, T, 2) → Rasterized Track Volume (T, H, W, C):

\[ \mathbf{c}[t,\, x_t^n,\, y_t^n] = v[n,t] \cdot \phi^n \qquad \phi^n \in \mathbb{R}^C \text{ (sinusoidal, C=64)},\quad v[n,t] \in \{0,1\} \]

Then patchified through the Track Projection Layer to match the video's spatiotemporal token grid.

Recommended data scale

StageScaleNotes
Stage 1 — Warm-up5k–20k pairsSmall heads converge quickly; backbone frozen
Stage 2 — In-domain50k–150k pairsOutput dim (64×T×2) is large — needs more data than typical OFT
Stage 3 — CycleNo extra dataSelf-supervised; reuses Stage 2 samples

4 · Architecture

VLT = Qwen3-VL backbone (unchanged) + three lightweight modules totalling <1% of backbone parameters.

Qwen3-VL backbone (original)

Input: text tokens + video frame tokens (T, H, W, 3)
         ↓  Visual Encoder (ViT)
       (T/2, H/28, W/28, d) video tokens
         ↓  3D-RoPE  — rotary PE over (t, h, w) per token
         ↓  Qwen3 LLM Transformer
         ↓  Language Head (LM head)
       text output  (VQA / captioning / QA)

Added modules

① Track Projection Layer

Input side. Maps rasterized track volume to track tokens.
Linear(14×14×2C → d)

② E_track (modality emb.)

Input side. Learnable \(\in \mathbb{R}^d\). Tells the transformer "this token is a track, not a video frame."

③ Track Regression Head

Output side. Projects final hidden state to target coordinates.
MLP(d → N×T×2)

Task special tokens

Input side. Learnable embeddings added to Qwen3-VL vocab for task routing and head selection.

Token flow

  <TASK_TOKEN> + text            video V_s (T,H,W,3)    track volume (T,H,W,C)
        │                               │                       │
        │                     Qwen3-VL ViT (frozen)    Track Proj. Layer ①
        │                               │                       │
        │                    video tokens (T/2,H/28,W/28,d)   track tokens
        │                               │   + E_video           │   + E_track ②
        │                               └──────── concat ────────┘
        └──────────────── Qwen3 LLM Transformer (LoRA) ──────────────────┘
                                         │
                        ┌────────────────┴──────────────────┐
                  Language Head                    Track Regression Head ③
               (Task B / C )                           (Task A')
                  text output                      T_t  (N, T, 2)

Why E_video + E_track?

3D-RoPE assigns the same positional encoding to a video token and a track token at the same (t, h, w). The modality embeddings add a learnable offset so the transformer can tell them apart in self-attention:

video_tokens = rope_encode(video_patches, t, h, w) + E_video
track_tokens = rope_encode(track_patches, t, h, w) + E_track   # NEW
tokens = concat([video_tokens, track_tokens], dim=seq)

Tokenization: shared spatial grid

Both video and track use the same patchify + merge pipeline, ensuring video_token[t,h,w] and track_token[t,h,w] share the same receptive field:

\[ (\cdot, H, W) \;\xrightarrow{\text{patchify}(2,14,14)}\; \xrightarrow{\mathbf{W}}\; \xrightarrow{\text{merge }2\times2}\; \left(\tfrac{T}{2},\tfrac{H}{28},\tfrac{W}{28}, d\right) \]

5 · Training

Stage 1
Track Module Warm-up

Backbone: fully frozen

Trained: ①②③ + task tokens

Loss: Task A' only

loss = huber(head(h), T_t_gt)
Stage 2
Multi-task Instruction Tuning

Backbone: LoRA or top-N unfrozen

Loss: A' + B + C mixed

\[ \mathcal{L} = 2\mathcal{L}_A + \mathcal{L}_B + \mathcal{L}_C + \mathcal{L}_M \]

Stage 3 (optional)
Cycle Consistency

Data: no new data needed

Use A'→B inverse to add a self-supervised signal:

\[ \mathcal{L}_{\text{cyc}} = \mathcal{L}_B\!\left(f_B(\hat{T}_t),\,\text{instr}\right) \]

7 · Summary

Three modules added to Qwen3-VL · <1% extra parameters · three-stage training · one representation \(R\) injected into the generator.

Architecture

Track Projection Layer ①, modality embedding E_track ②, Track Regression Head ③ on top of a frozen Qwen3-VL backbone. Shared 3D-RoPE grid ensures pixel-level video–track alignment.

Training

Multi-task over A' / B / C / C-align teaches Object Grounding, Motion Semantics, and Caption-Track Alignment from a single backbone.

Value

The representation \(R\), injected into the generator's attention, makes the generator treat C_t and T_t as a single coherent semantic rather than independent conditions.
