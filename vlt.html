1 · Motivation & Goal

1.1 The Full Pipeline

The goal is to inject high-level reasoning into a track-conditioned video-to-video generator.

The base generator takes the following inputs:

Input: V_s (source video), T_s (source track), T_t (target track), C_t (target caption)
          ↓
     [ Track-conditioned Video Generator ]
          ↓
Output: V_t (generated target video)
  • V_s — carries appearance and scene structure, but the generator sees it only as a pixel-level signal.
  • T_s, T_t — given as track coordinate sequences, but their meaning (which object, what kind of motion) is not understood.
  • C_t — a semantic description of the target video, but its relationship to T_t is never explicitly modeled.

VLT learns the relationships among scene, motion, and caption expressed by these four signals, and injects the resulting representation \(R\) into the generator's attention layers so that the generator produces more coherent and semantically meaningful video.

1.2 Limitations of the Generator

  1. No semantic understanding of V_s — it sees V_s as raw pixels and does not know which objects exist or how the scene is structured.
  2. No understanding of T_s→T_t motion semantics — it can follow coordinate changes but does not conceptually grasp whether the motion is camera vs. object motion, zoom, rotation, etc.
  3. No cross-modal alignment between C_t and T_t — C_t says "what / how to show," T_t says "where / how to move," but the generator never understands that these are two expressions of the same physical event.

1.3 Three Capabilities VLT Must Provide

  1. Object & Scene Grounding — understand which objects exist in V_s and which object/region a given track belongs to.
    e.g. "this track is the person's head," "this track is a background tree."
  2. Motion Semantics Understanding — understand what kind of motion the T_s→T_t change represents (camera / object / zoom / rotation), and in what direction and speed.
    e.g. "camera pans right," "object moves forward quickly," "the whole scene zooms in."
  3. Caption-Track Alignment — judge which phrase of C_t (e.g. "runs to the right") corresponds to which motion pattern in T_t, and whether the two are consistent.
    e.g. "C_t says the person moves right, but T_t moves left → mismatch."

The representation \(R\) produced by VLT compresses the information from these three capabilities; the generator consumes it (via cross-attention) to produce video where scene, motion, and caption are mutually consistent.

2 · Training Task Definition

VLT is a multi-task model that shares a single backbone and common modules while performing several tasks simultaneously. This section defines what learning signal each capability receives from each task.

2.1 Task Special Token Design

Each task is distinguished by a special token prepended to the input text. These are learnable tokens newly added to Qwen3-VL's embedding table, serving two roles:

  1. Task routing — telling the model which task to perform.
  2. Head selection — deciding whether to use the Track Regression Head (Task A') or the Language Head (Task B/C).
Special TokenTaskOutput Head
<PREDICT_TRACK>Task A' — track coordinate predictionTrack Regression Head
<DESCRIBE_MOTION>Task B — motion description generationLanguage Head
<MOTION_QA>Task C — motion-related QALanguage Head
<MATCH_CAP_TRACK>Task C-align — caption–track matchingLanguage Head (or binary head)

Usage examples:

<PREDICT_TRACK> Source caption: ... Target caption: ...
<DESCRIBE_MOTION> Describe how the motion changes between the source and target tracks.
<MOTION_QA> Question: Is this motion caused by the camera or the object? Answer concisely.
<MATCH_CAP_TRACK> Does the target caption match the motion of the target track? Answer Yes or No.

2.2 Task A' — PREDICT_TRACK (Motion Reasoning)

Goal: Accurately predict the source→target motion at the coordinate level, jointly learning motion semantics and grounding.

  • Input: <PREDICT_TRACK> + (V_s, T_s, C_t) or (V_s, T_s, motion_instruction)
  • Output: T_t (N×T×2)
  • Loss: Huber/L1 loss on coordinates

To solve this task VLT must (a) recognize in V_s the object/region the track follows (Object & Scene Grounding), and (b) understand how the motion described by C_t or the instruction applies to which part of T_s (Motion Semantics + part of Caption-Track Alignment).

2.3 Task B — DESCRIBE_MOTION (Motion → Language)

Goal: Describe track changes and scene context in language, aligning motion semantics into the language space.

  • Input: <DESCRIBE_MOTION> + (V_s, T_s, T_t)
  • Output: motion description text
  • Loss: Cross-entropy on generated tokens

Example outputs:

  • "The person moves quickly to the right while the background stays static."
  • "The camera zooms in while the object remains centered."

2.4 Task C — MOTION_QA (three sub-types)

Goal: Condition on all of V_s, T_s, T_t, C_t to answer compound motion-related questions, reinforcing the three capabilities at a fine-grained level.

  • Input: <MOTION_QA> + (V_s, T_s, T_t, C_t, Q)
  • Output: A (text or multiple choice)
  • Loss: Cross-entropy

Question templates fall into three categories:

  1. C-obj (Object Grounding) → reinforces Object & Scene Grounding
    • "What object does the moving track belong to?"
    • "Is the moving object the person or the ball?"
  2. C-motion (Motion Semantics) → reinforces Motion Semantics Understanding
    • "Is this motion caused mainly by the camera or the object?"
    • "Does the object speed up or slow down?"
    • "In which direction does the object move at the end?"
  3. C-align (Caption-Track Alignment) → directly trains Caption-Track Alignment
    • "Does the target caption correctly describe the motion of the track?" (Yes/No)
    • "Which phrase in the caption corresponds to the target track motion?"

2.5 Capability ↔ Task Mapping

CapabilityPrimary Training Tasks
Object & Scene GroundingTask A', Task C-obj
Motion Semantics UnderstandingTask A', Task B, Task C-motion
Caption-Track AlignmentTask A' (C_t condition), Task C-align, C-align-style QA

3 · Data Design

3.1 Sample Structure

Each training sample is a 6-tuple:

(V_s, T_s, C_s, V_t, T_t, C_t)

V_s: source video  (T, H, W, 3)
T_s: source track  (N=64, T, 2)   — pixel coordinates (x, y)
C_s: source caption (text)
V_t: target video  (T, H, W, 3)
T_t: target track  (N=64, T, 2)   — points corresponding to T_s
C_t: target caption (text)

Source and target are a pair with identical appearance/background but different motion. The N=64 track points are random pixels sampled from the video and tracked with a tracker such as CoTracker.

3.2 Automatic Instruction Generation

The already-available C_s and C_t are fed to an LLM to automatically generate a motion instruction:

C_s: "A person walking slowly to the left in a park"
C_t: "A person running quickly to the right in a park"
       ↓  LLM
Instruction: "Make the person move faster and change direction to the right"

As a fallback — when C_s and C_t are nearly identical or motion information is insufficient — analyze ΔT = T_t − T_s to generate a templated sentence.

3.3 Track Volume Conversion

Convert the sparse track (N, T, 2) into a Rasterized Track Volume (T, H, W, C):

\[ \mathbf{c}[t,\, x_t^n,\, y_t^n] = v[n,t] \cdot \phi^n \]

  • \(\phi^n \in \mathbb{R}^C\): random sinusoidal embedding of the n-th point (C=64)
  • \(v[n,t]\): visibility mask (0 or 1)
  • all other positions: zero

It is then patchified and passed through the Track Projection Layer to become tokens on the same spatiotemporal grid as the video.

3.4 Recommended Data Scale

StagePurposeRecommended ScaleRationale
Stage 1 (Warm-up)Initialize track modules5k–20k video pairsEnough for small heads to converge (VLM4VLA, OFT regime)
Stage 2 (in-domain)Track prediction + description50k–150k video pairsOutput dim (64×T×2) is large, so slightly larger than OFT

4 · VLT Architecture

4.1 Qwen3-VL Backbone Recap

VLT uses Qwen3-VL directly as the backbone. The original structure:

[ Original Qwen3-VL ]

Input: text tokens + video frame tokens
         ↓
  Visual Encoder (ViT-based)
  (T, H, W, 3) → (T/2, H/28, W/28, d) tokens
         ↓
  3D-RoPE: rotary PE over (t, h, w) axes per token
         ↓
  Qwen3 LLM Transformer
         ↓
  Language Head (LM head)
         ↓
  Output: text tokens (VQA, captioning, ...)

i.e. the original Qwen3-VL takes only text + video tokens and performs video understanding / description / QA.

4.2 VLT = Qwen3-VL + Minimal Modules

VLT adds only the minimal modules needed to process tracks on top of Qwen3-VL.

[ VLT Architecture ]

Input:
  [<TASK_TOKEN> + text tokens]            ← Task special token added
  [source video V_s (T,H,W,3)]
       ↓ Qwen3-VL ViT (existing)
  video tokens  (T/2, H/28, W/28, d)
                                          ← identical to Qwen3-VL up to here
  [track volume (T,H,W,C)]
       ↓ Track Projection Layer           ← NEW ①
  track tokens  (T/2, H/28, W/28, d)

  ─────────────────────────────────────
  token sequence:
    tokens = concat(
        video_tokens + E_video,           ← E_video: existing or new modality embedding
        track_tokens + E_track,           ← E_track: NEW ②
        dim=sequence
    )
  ─────────────────────────────────────

  Qwen3 LLM Transformer (existing, LoRA fine-tuning)
         ↓
  hidden states h ∈ ℝ^(seq_len, d)
         ↓
  ┌──────────────────┴──────────────────┐
[Language Head]               [Track Regression Head]
(reuse existing LM head)       (NEW ③, MLP)
       ↓                               ↓
  text output                  target track (N, T, 2)
  (Task B, C, C-align)         (Task A')

Added modules

ModuleLocationRole
① Track Projection Layerinput sidetrack volume → track tokens, Linear(14×14×2C, d)
② E_track (modality embedding)input sidelearnable vector distinguishing video vs. track tokens
③ Track Regression Headoutput sidehidden state → target track coords, MLP(d → N×T×2)
Task special tokensinput sidetask routing signal (<PREDICT_TRACK> etc.)

Total added parameters are at the <1% level of the backbone.

4.3 Modality Embedding: E_video & E_track

E_video and E_track are learnable vectors \(\in \mathbb{R}^d\) added to video tokens and track tokens respectively.

Why this is needed: Qwen3-VL encodes position via 3D-RoPE, so a video token and a track token at the same (t, h, w) share the same RoPE. Their position is distinguishable, but their modality is not encoded by RoPE alone. E_video and E_track encode this modality identity.

# Positional Encoding: same (t,h,w) → same RoPE
video_tokens = rope_encode(video_patch_tokens, t, h, w)  # existing
track_tokens = rope_encode(track_patch_tokens, t, h, w)  # same PE applied

# Modality Embedding: distinguish modality
video_tokens = video_tokens + E_video   # E_video ∈ ℝ^d, learnable
track_tokens = track_tokens + E_track   # E_track ∈ ℝ^d, learnable (NEW)

# Sequence: concat, not summed
tokens = concat([video_tokens, track_tokens], dim=sequence)

This makes it clear, in transformer self-attention, which modality information is being attended to when a video token attends to a track token or vice versa.

4.4 Video & Track Tokenization

Qwen3-VL tokenizes (T,H,W,3) as:

\[ (T, H, W, 3) \xrightarrow{\text{patchify (2,14,14)}} \left(\tfrac{T}{2}, \tfrac{H}{14}, \tfrac{W}{14}, 1176\right) \xrightarrow{\text{Linear}} \left(\tfrac{T}{2}, \tfrac{H}{14}, \tfrac{W}{14}, d\right) \xrightarrow{\text{spatial merge 2×2}} \left(\tfrac{T}{2}, \tfrac{H}{28}, \tfrac{W}{28}, d\right) \]

The same grid is applied to the track volume, matched in dimension by the Track Projection Layer Linear(14×14×2C → d), then merged identically:

\[ (T, H, W, C) \xrightarrow{\text{patchify (2,14,14)}} \left(\tfrac{T}{2}, \tfrac{H}{14}, \tfrac{W}{14}, 14{\times}14{\times}2C\right) \xrightarrow{\mathbf{W}_\text{track}} \left(\tfrac{T}{2}, \tfrac{H}{14}, \tfrac{W}{14}, d\right) \xrightarrow{\text{merge}} \left(\tfrac{T}{2}, \tfrac{H}{28}, \tfrac{W}{28}, d\right) \]

This way video_token[t, h, w] and track_token[t, h, w] share the same receptive field, guaranteeing pixel-level alignment.

5 · Training

5.1 Stage 1 — Track Module Warm-up

  • Backbone fully frozen
  • Train only Track Projection Layer, Track Regression Head, E_track, and task special tokens
  • Loss: Task A' Huber loss only
loss_A = huber_loss(track_reg_head(h_last), T_t_gt)

5.2 Stage 2 — Multi-task Instruction Tuning

  • Backbone: LoRA or unfreeze top N layers
  • Loss: mixture of Task A'/B/C and C-align

\[ \mathcal{L} = \lambda_A \mathcal{L}_{\text{track}} + \lambda_B \mathcal{L}_{\text{desc}} + \lambda_C \mathcal{L}_{\text{VQA}} + \lambda_M \mathcal{L}_{\text{match}} \]

Recommended ratio: \(\lambda_A : \lambda_B : \lambda_C : \lambda_M = 2 : 1 : 1 : 1\)

5.3 Stage 3 — Cycle Consistency (Optional)

Exploit the inverse relationship between Task A' and B to add a self-supervised signal:

\[ \mathcal{L}_{\text{cycle}} = \mathcal{L}_{\text{desc}}\big(f_B(\hat{T}_t), \text{instr}\big) \]

6 · Experimental Plan

6.1 Evaluation Metrics

  • Task A': per-point L2/Huber error, AEE, Occlusion Accuracy
  • Task B: BLEU-4, CIDEr, BERT-score (semantic similarity)
  • Task C: Top-1 QA accuracy

6.2 Capability Verification Experiments

  • Instruction Sensitivity Test — does T_t change when different instructions are given for the same (V_s, T_s)?
  • Caption-Track Consistency Test — internal activation difference when C_t ↔ T_t match vs. mismatch.
  • Track Ablation — measure performance drop when the track is removed (does the track actually contribute?).
  • Cycle Consistency — semantic similarity of instruction regeneration along Task A' → Task B.
  • Downstream Quality — SSIM/LPIPS/FVD of a generator with injected VLT representation R vs. the base generator without it.

7 · Summary

VLT is built by adding just three lightweight modules on top of Qwen3-VL — Track Projection Layer (①), Track Modality Embedding E_track (②), and Track Regression Head (③).

Tracks are processed as separate tokens on the same spatiotemporal grid as the video; sharing 3D-RoPE plus modality embeddings achieves both pixel-level video–track alignment and modality distinction simultaneously.

Through multi-task training over Task A'/B/C and C-align, VLT learns the three capabilities — Object & Scene Grounding, Motion Semantics Understanding, and Caption-Track Alignment — and the final representation \(R\) encodes, at a high level, the relationships among the four conditions (V_s, T_s, T_t, C_t).

The core value of VLT is that this \(R\), injected into the attention layers of the track-conditioned video generator, helps the generator treat C_t and T_t not as independent conditions but as a single, coherent semantic.
